Technical Build Spec

From clickable prototype to a real, logged-in product.

The current Training Hub is a polished React front-end running entirely on mock data. This document lays out how to make it real: the stack, the front-end code structure, a backend with a proper database, media storage, live-session realtime, and a login system with coach / admin / parent roles.

React + Vite front-end Supabase backend Postgres + Row-Level Security ~£0–25 / month to run
Where we are

What exists today, and what's missing

The prototype already proves out the whole product surface. Everything below the glass is faked — and that's exactly the gap this spec fills.

✓ Already built (front-end)

  • Drill Library, filtering & drill detail pages
  • Session Planner & saved Sessions
  • Templates and Media Library screens
  • Live / run-session full-screen mode
  • Brand, dark mode, responsive + mobile nav

⚠ Faked / missing

  • All data lives in data.js — nothing persists
  • No accounts, no login, no "my sessions"
  • Media isn't really uploaded or stored
  • Nothing is shared between coaches
  • No server, no database, no API
The core idea: keep the React front-end almost exactly as it is. Replace the single window.OTJ mock object with a thin data layer that talks to a hosted backend. Add a login screen in front. That's the whole job — everything else is filling in those two boxes.
The tools

Recommended stack

Optimised for a volunteer-run grassroots club: cheap, low-maintenance, no dedicated ops, and a single managed service that covers database + auth + file storage + realtime.

Front-end
React + Vite + TypeScript

You already have the React components. Move them off the in-browser Babel setup into a Vite project for a real build, routing and types. react-routertanstack-query

Backend
Supabase

One managed platform giving you a Postgres database, auto-generated REST + realtime APIs, authentication, and file storage. Removes the need to write/host a server for v1. postgresauthstorage

Database
PostgreSQL

Relational data fits perfectly — drills, sessions and templates are all linked records. Row-Level Security enforces "who can see what" at the database itself.

Auth
Supabase Auth

Email + password and magic-link out of the box, JWT sessions, password reset, and roles via a profiles table. No auth code to write or secure yourself.

Hosting
Vercel / Netlify

Push the front-end to a static host with a free tier and automatic HTTPS. Backend is hosted by Supabase.

Why not a custom Node/Express server? You can, and the API section below maps cleanly onto one if you outgrow Supabase. But for a club app with a few dozen coaches, a managed backend gets you to "real and secure" in days, not weeks — and there's no server to patch at 11pm.
The shape of it

How the pieces fit together

Three tiers. The browser holds the UI and a logged-in session; the managed backend holds data, files and the rules about who can touch them.

The React app never talks to the database directly — every request carries the logged-in user's token, and Postgres' Row-Level Security decides what that user is allowed to read or write. There's no way to "forget" an auth check because the rule lives on the data, not in the UI.

Front-end

Code structure

Lift the existing screens into a proper Vite project. The component split you already have (home / library / sessions / media / live) becomes your route map almost 1:1.

The migration, concretely

Scaffold a Vite + React + TS project

Drop your styles.css in as-is — the design tokens carry over untouched. The fonts and brand don't change at all.

Move each screens-*.jsx into routes/

Wire them with react-router so the URL reflects the screen (today routing is hand-rolled in app.jsx state). Bookmarkable /drill/d6 links come for free.

Replace window.OTJ reads with query hooks

Every OTJ.drills becomes useDrills(); every upsertSession() becomes a mutation. The components barely change — only where the data comes from.

Wrap everything in an auth guard

No session → render <Login/>. Session present → render the app. The role decides which nav items show.

Front-end

Data & state

Use TanStack Query as the single source of truth for server data — it handles caching, loading states, and refetching so you don't hand-roll any of it. Local UI state (which filter is open, dark mode) stays in React state as it is now.

// lib/queries.ts — reads and writes in one place
export const useDrills = (filters) =>
  useQuery({
    queryKey: ['drills', filters],
    queryFn: () => supabase
      .from('drills')
      .select('*, media:media_id(*)')   // joins the clip
      .match(filters)
  });

export const useSaveSession = () => {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (s) => supabase.from('sessions').upsert(s),
    onSuccess: () => qc.invalidateQueries(['sessions'])
  });
};
Server data

Drills, sessions, media, templates → TanStack Query. Cached, auto-refetched.

Session / user

A small useAuth() context exposing the user + role app-wide.

UI state

Filters, modals, dark mode → plain useState, same as today.

Backend

API design

With Supabase the REST API is generated from your tables, so this is really a list of operations rather than hand-written endpoints. If you build a custom server instead, these map directly to routes.

Drills & content

  • GET  /drills — list + filter by corner, age, skill
  • GET  /drills/:id — one drill + its media
  • POST /drills — coach adds a drill
  • PATCH /drills/:id — edit
  • GET  /templates — shared session templates

Sessions & media

  • GET  /sessions?coach=me — my plan
  • POST /sessions — save from planner
  • POST /media — register an uploaded file
  • GET  /media — library, with usage counts
  • POST /storage — get a signed upload URL

Filtering, ordering and pagination are query params on the same endpoints — exactly the filters the Library screen already exposes (corner, age band, skill, level).

Backend

Media storage

The Media Library's "Upload media" button needs real file storage. Video, images and PDFs go to Supabase Storage; the database just keeps a pointer and metadata.

Coach picks a file

Front-end asks the backend for a short-lived signed upload URL — the file never passes through your own server.

Browser uploads straight to Storage

With a progress bar. Large match clips don't choke an API server because there isn't one in the path.

Register a media row

Save name, type, size, dimensions and the storage path. This is the record the Library and drill pages read.

Serve via signed URLs

Playback/preview uses time-limited signed links, so only logged-in club members can view files.

YouTube items store just the URL — no upload, no storage cost. Diagrams and PDFs follow the same path as video.

Backend

Live session — the realtime bit

The full-screen run-session mode works solo today. The valuable upgrade: a coach running the session on the touchline and an assistant following along on another phone, in sync.

v1 — solo (no backend needed)

Timer and current-activity state live in the browser. Persist progress to localStorage so a refresh mid-session doesn't lose the place — the prototype pattern, kept.

v2 — shared (Supabase Realtime)

The session broadcasts "now on activity 3, 4:12 remaining" over a Realtime channel. Other devices subscribe and mirror it. Same WebSocket tech already in the stack — no extra service.

Accounts

Login & authentication

A new Login screen becomes the front door. Supabase Auth handles the hard, security-sensitive parts; you build the screen and the guard.

The login flow

Coach enters email + password

Or requests a magic link — a one-tap sign-in email, ideal for less technical volunteers. Both are built into Supabase Auth.

Backend verifies & returns a JWT

A signed token proving who they are, stored by the client library. You never see or store the raw password.

App loads their profile + role

useAuth() reads the profiles row → name, club and role. The sidebar coach-chip and visible nav are driven by this.

Every request carries the token

The data layer attaches it automatically. Postgres Row-Level Security checks it on every read and write.

Session persists & refreshes

Reopen the app → still logged in. Tokens auto-refresh. "Log out" clears it. Password reset is a built-in email flow.

// routes/Login.tsx — the entire auth call
const { error } = await supabase.auth
  .signInWithPassword({ email, password });

// App.tsx — the guard
const { user, loading } = useAuth();
if (loading) return <Splash/>;
if (!user)   return <Login/>;
return <AppShell/>;   // the hub you already built
Sign-up is invite-only. A club app shouldn't let strangers self-register. The admin invites a coach by email; they set a password via the link. This keeps the drill library and member data inside the club.
Accounts

Roles & permissions

Three roles cover the club. The role on each profile drives both what the UI shows and — crucially — what the database lets them do, enforced by Row-Level Security so the rules can't be bypassed from the front-end.

Coach coach

The everyday user — e.g. Sarah Thompson, U8s.

  • Browse the full club drill library & templates
  • Create, edit and run their own sessions
  • Add drills & upload media to the shared library
  • Cannot edit another coach's sessions or manage accounts

Admin / Dev Manager admin

Runs the club's coaching programme.

  • Everything a coach can do, across all age groups
  • Invite & remove coaches, set roles
  • Curate official templates and the drill library
  • Manage club branding & settings

Parent / Player parent — optional, later

Read-only window into their child's team.

  • See upcoming sessions: when, where, the focus
  • View "what we're working on" drills with video
  • No planning, no editing, no library management
-- Row-Level Security: a coach only writes their own sessions
CREATE POLICY "own sessions" ON sessions
  FOR ALL USING ( coach_id = auth.uid() );

-- everyone in the club can read the shared drill library
CREATE POLICY "club drills" ON drills
  FOR SELECT USING ( club_id = my_club() );
Ship it

Hosting & running cost

How it's deployed

  • Front-end → Vercel/Netlify, auto-deploys on git push, free HTTPS
  • Backend → Supabase project, fully managed
  • Domain → e.g. hub.ossetttownjnr.com
  • Backups → automatic daily Postgres backups

Realistic monthly cost

  • Free to start — both tiers cover a small club comfortably
  • ~£20–25/mo once media storage grows (Supabase Pro)
  • No server to maintain, patch or pay for separately
  • Scales to many teams before cost becomes a factor
Ship it

Build roadmap

Ship something real fast, then layer on the collaborative features. Each phase is independently useful.

Phase 1~1 week
Real project + login

Vite migration, Supabase set up, the six tables, login screen + auth guard. The app runs on real (seeded) data behind a password.

Phase 2~1 week
Persisted planning

Drills, sessions and templates read/write to the database. The planner saves for real. Row-Level Security locks each coach to their own sessions.

Phase 3~3–4 days
Media uploads

Signed-URL uploads to Storage, the Media Library backed by real files, drills linked to real clips.

Phase 4~3–4 days
Admin & invites

Invite-only sign-up, the admin role, coach management, official template curation.

Phase 5later
Realtime & parents

Shared live-session sync, the read-only parent role, push reminders for upcoming sessions.

Bottom line: the expensive, design-heavy work — the entire product UI — is done. What's left is well-trodden plumbing: a managed backend, a thin data layer, and a login screen. A capable developer gets to a logged-in, persistent v1 in roughly two weeks.
